第一层：按流程展开

SPEC 修订总量 = SPEC 数量 × 平均修订轮数 即时总耗时 = SPEC 修订总量 ×（SPEC 编辑 + AI 方案处理 + 一次代码落地 + 方案审查） 邮件总耗时 = 平均修订轮数 ×［SPEC 数量 × SPEC 编辑 + AI 方案处理 + SPEC 数量 × 方案审查］+ 一次代码落地 文档总耗时 = SPEC 修订总量 × SPEC 编辑 + 文档中未被覆盖的 AI 等待 + SPEC 修订总量 × 方案审查 + 一次代码落地

即时通讯的方案和代码在每轮中交错发生；邮件和文档则在方案确认后只落地一次代码。

第二层：提取共同项

共同基础成本 = SPEC 修订总量 × SPEC 编辑 + SPEC 修订总量 × 方案审查 + 一次代码落地 即时总耗时 = 共同基础成本 + SPEC 修订总量 × AI 方案处理 +（SPEC 修订总量 − 1）× 一次代码落地 邮件总耗时 = 共同基础成本 + 平均修订轮数 × AI 方案处理 文档总耗时 = 共同基础成本 + 文档中未被覆盖的 AI 等待

这组公式只用于横向比较。模式开销等于模式总耗时减去共同基础成本，并不一定代表无效工作。

调度假设与边界

平均 1.5 轮包含首轮。即时与文档把它解释为每个 SPEC 的平均修订轮数；邮件把它解释为整个批次的平均往返轮数。

文档协作先连续完成所有首轮编辑，同时增量提交 AI；之后优先审查最早完成的结果，需要修订时立即编辑并再次提交。模拟器只为文档模式枚举 SPEC 的第二轮组合，邮件不会沿用这个概率模型。

  • 开发者只有一个编辑/审查执行槽，AI 任务没有并发数量上限。
  • 当前只支持 1 到 2 轮；第二轮概率等于“平均修订轮数 − 1”。
  • 代码落地是一个复合输入，可以包含希望纳入比较的代码检查、合并和测试。
  • 不计算现实邮件的离线查收延迟、部署发布和模式交接成本。

该工具比较的是给定假设下的墙钟时间，不是对所有团队和工程任务都成立的性能承诺。
