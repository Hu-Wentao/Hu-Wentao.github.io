一个项目经过几轮 Vibe Coding 之后，通常不会缺少文档。真正麻烦的是文档越来越多，却没有人能明确回答：哪一份描述当前事实，哪一份只是未来目标，哪一个测试足以证明需求已经交付，哪一份旧设计已经失去约束力。

代码说功能已经上线，计划还留着待办项；E2E 已经通过，需求状态仍然是 Planned；架构文档描述的是目标方案，Agent 却把它当作当前实现继续开发。人类可以凭借会议记忆和项目经历勉强分辨这些差异，刚进入项目的 Agent 只能从互相冲突的文件中猜一个答案。

为此，我实现了 [project-governance](https://github.com/Hu-Wentao/skills/tree/main/skills/project-governance) Skill。它不是用于生成更多文档的模板，而是一套面向长期演进的项目事实治理方法：先区分每条信息属于哪个权威层，再管理需求、基线、计划和验证证据之间的生命周期，遇到语义冲突时将决定权明确交还给产品负责人。

它的核心可以概括成一句话：

> 不要求所有文档说同一句话，而是让每类文档只对自己负责的事实拥有权威。

## 六种事实，六种职责

`project-governance` 要求每条重要信息只归属于一个主要权威层：

| 权威层 | 回答的问题 | 适合保存的内容 |
| --- | --- | --- |
| Requirement | 用户最终需要什么 | 稳定 ID、参与者、用户结果、产品约束、可观察验收 |
| Baseline | 当前有哪些长期有效的规则 | 服务边界、权限、数据保证、信任边界、兼容承诺 |
| Plan | 准备如何到达目标 | 目标设计、迁移阶段、临时方案、未决问题、完成条件 |
| Code and test fact | 系统现在具体如何实现 | Schema、API、算法、文件位置、运行行为和测试细节 |
| Archive | 过去为什么这样决定 | 已替代、已拒绝或已完成且不再约束未来的设计材料 |
| Operational workflow | 一项工作怎样被安全重复执行 | 项目命令、环境拓扑、发布和恢复流程、项目级 Skill |

这个分类看起来简单，却解决了文档协作中最常见的混乱：把未来方案写成当前架构，把一次实现细节写成永久需求，把通过测试直接等同于产品状态，或者让已经归档的设计继续影响新代码。

例如，“用户可以导出自己的账户数据”是需求；“导出任务只能由账户所有者创建，并保留可审计记录”是当前基线；“将同步导出迁移为异步任务”是计划；队列表名、接口路径和重试算法则属于代码与测试事实。它们互相关联，但不应该复制成四份措辞略有差异的真相。

## 为什么项目需要 Baseline

需求描述用户结果，代码描述当前实现，但两者之间仍然缺少一层：即使未来替换框架、数据库或服务拓扑，也必须继续成立的工程与产品约束。

这正是 Baseline 的职责。它保存已经生效、并且会约束后续实现的稳定规则，例如：

- 哪个组件拥有某项职责；
- 谁可以执行某个操作；
- 哪些数据不可修改或只能追加；
- 密钥允许流经哪些边界；
- 客户端可以依赖怎样的兼容承诺；
- 哪些网络表面属于公开、内部或管理入口。

Baseline 不应该成为数据库表、API 和文件路径的快照。这些细节发生变化时，代码和测试本身才是更准确的事实源。Baseline 只保留跨实现仍然重要的决定。

同样，尚未完全生效的目标不能提前写进 Baseline。如果异步导出只完成了一半，计划仍应保持 Partially implemented；最多将已经稳定生效的子集写入 Baseline，并明确剩余差距。否则下一位 Agent 会依据一份“看起来已经完成”的架构继续工作，最终制造更隐蔽的偏差。

## 文档不是写完，而是进入生命周期

很多项目按时间归档文档：文件足够旧，就移动到 `archive/`。`project-governance` 使用的是语义标准：只有当一份文档不再约束未来工作时，它才应该归档。

一个计划完成后，需要经历下面的状态转换：

~~~text
计划中的目标设计
    ↓ 实现完成
检查代码、验收证据与兼容行为
    ↓
把仍需长期成立的规则合入 Baseline
    ↓
单独评审 Requirement 是否可以变为 Active
    ↓
归档已失去约束力的实施计划，并修复索引与链接
~~~

这里故意把“实现完成”“更新 Baseline”“激活 Requirement”和“归档 Plan”拆成不同动作。测试通过只说明某些行为在特定环境中得到验证，不会自动证明所有负向约束、兼容承诺和运行控制都已经满足。

如果计划只交付了一部分，就继续保留计划，并准确写明哪一部分已经成为当前事实。既不把整个目标提前提升为 Baseline，也不因为项目时间很久就将未完成工作扔进 Archive。

## 稳定 ID 保护需求的历史含义

需求需要稳定 ID，不只是为了方便全文搜索，而是为了让设计、代码、测试和决策记录引用同一个长期语义。

当修改只是澄清措辞，且用户结果没有变化时，可以保留原 ID；当参与者、结果、权限、经济含义、数据保证或验收语义发生实质变化时，应创建新 ID，并保留旧 ID 的替代关系或废弃原因。已经发布的 ID 永远不应被重新分配给另一个需求。

例如：

```md
### REQ-ACCOUNT-001 Export account data

- Status: Planned
- Actor: Account owner
- Goal: Export a portable copy of personal account data
- Acceptance: The owner can request and retrieve an export without exposing another account's data
```

将“导出文件在 24 小时内可下载”改写为更清晰的等价表述，可能只是澄清；将导出权限扩展给组织管理员，则改变了参与者与权限语义，不能为了保持编号连续而悄悄覆盖原需求。

Skill 因此要求修改需求前先找到所有引用，判断这是澄清、缩小、扩展、替代、拆分、合并、状态变化还是优先级变化，再同步检查关联 Baseline、计划、实现和验收证据。优先级和产品状态始终是产品决定，不能从实现成本、测试数量或文档年龄中推断。

## 测试覆盖不是越多越好

需求与测试之间需要可追溯，但这并不意味着每条验收规则都要在 E2E、浏览器测试和单元测试中重复一遍。`project-governance` 为每种行为选择能够有效证明它的最小验证归属：

| 行为 | 主要验证归属 |
| --- | --- |
| 跨服务、用户可见的完整流程 | API、集成或 Docker E2E |
| 只发生在 UI 的交互与渲染 | 浏览器或 UI 测试 |
| 算法、输入校验、权限不变量 | 聚焦的单元或集成测试 |
| 部署、密钥轮换、恢复和运行时访问 | 人工或运维验证 |
| 可替换的实现细节 | 代码附近的聚焦测试 |

多层测试只有在分别证明不同风险时才有价值。把所有细节塞进一个跨服务 E2E，会让测试既慢又脆弱；只用 API 测试证明 UI 行为，或者用一次成功部署证明密钥轮换策略，也都选错了证据所有者。

可追溯记录应该保存需求 ID、主要验证归属、必要的辅助证据和已知缺口。具体命令、某次运行时间和日志属于测试或交付产物，不应该不断写回稳定需求正文。

## 冲突时，Agent 不应该替产品做决定

项目治理并不追求让所有文件表面一致。不同权威层发生冲突时，Skill 会先列出互相矛盾的主张及其来源，再判断它属于哪一种问题：

- Requirement 与 Baseline 冲突，可能是产品决定发生变化，也可能是 Baseline 过期；
- Baseline 与代码冲突，可能是实现缺陷，也可能是文档没有跟上；
- Plan 与代码不同，可能只是计划尚未完成；
- Requirement 与测试不同，可能缺少证据，也可能验收已经过时；
- Archive 与当前资料不同，Archive 自动失去权威。

当语义没有歧义时，Agent 可以在获得编辑授权后修复断链、重复 ID 等机械漂移。但如果修复会改变用户结果、权限、数据保证、兼容性、优先级或验收含义，它必须停止并请求产品决定，而不是选择最方便实现的一种解释，再把其他文件一起改到“看起来一致”。

## 通用治理与项目 Skill 的边界

治理方法可以跨项目复用，具体操作知识却通常不能。`project-governance` 只提供通用的权威分层、生命周期和审查规则；确切命令、环境名称、服务拓扑、端口、领域术语、项目安全边界和恢复流程应该留在项目级 Skill 中。

并非每条约定都值得制作成 Skill。一项流程只有在重复执行、容易因顺序错误造成损失、跨越多个工具、需要反复重新发现项目知识，或适合用确定性脚本编码脆弱步骤时，才值得提炼。一次性计划、普通编码规范、单条命令和仍不稳定的产品提案，放在项目文档或仓库说明中通常更合适。

这让通用 Skill 保持轻量，也避免将某个项目的业务决定错误地带入另一个项目。

## 自动检查结构，但不自动裁决语义

Skill 自带一个 Markdown 治理文档校验器，可以检查重复 Requirement ID、引用了但未声明的 ID、本地断链，以及计划目录是否缺少状态索引：

```bash
node /path/to/project-governance/scripts/validate-governance.mjs \
  --root /path/to/project
```

如果治理文档不在默认的 `docs/` 下，可以增加 `--docs` 指定相对或绝对路径。

这个脚本只处理确定性结构缺陷。它不会判断某条需求应该是 Planned 还是 Active，不会猜测某个计划是否真正完成，也不会替产品负责人决定优先级。结构可以自动验证，语义仍然必须由证据和明确授权共同决定。

## 安装与使用

可以通过 `skills` CLI 从我的 Skill 集合中安装：

```bash
pnpm dlx skills add Hu-Wentao/wyatt_skills \
  --skill project-governance \
  --yes
```

安装后，可以让 Agent 从一次只读审查开始，不必立刻重组整个项目：

```text
请使用 project-governance Skill 审查当前项目文档。
先保持只读，识别 Requirement、Baseline、Plan、代码与测试事实、Archive
和项目操作流程各自的权威来源；列出冲突、验证缺口和建议的最小治理结构，
不要为了统一措辞而修改产品语义。
```

也可以让它治理一条具体需求：

```text
请使用 project-governance Skill 审查 REQ-ACCOUNT-001。
判断本次修改属于澄清还是语义变化，查找所有引用，核对关联 Baseline、
实现和验证证据；如果会改变参与者、权限或验收含义，先请求产品决定。
```

## 结语

Vibe Coding 让代码和文档都能被更快地生成，也让项目更容易同时存在多份似是而非的“真相”。解决办法不是要求 Agent 每次读完所有历史，更不是再增加一份包罗万象的总文档，而是建立清晰的事实边界和生命周期。

Requirement 负责用户结果，Baseline 负责当前长期规则，Plan 负责尚未完成的目标，代码和测试负责实现事实，Archive 保留历史，项目级 Skill 保存可重复执行的操作知识。当每一层只对自己的问题拥有权威，Agent 才能知道什么可以直接修复，什么需要继续验证，什么必须等待人类做出产品决定。

项目治理最终不是为了把文档整理得更漂亮，而是让人和 Agent 在几个月后的下一轮修改中，仍然能够分清项目从哪里来、现在是什么，以及接下来准备去哪里。
