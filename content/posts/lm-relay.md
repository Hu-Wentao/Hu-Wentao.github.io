最近一段时间, 我越来越频繁地在不同模型服务之间切换. 有些请求会发到本地 `LM Studio`, 有些需要走远程 OpenAI 兼容接口, 还有一些则依赖额外的 `cli` provider。表面上看, 这只是多写几个 `base_url` 和 `api_key` 的问题, 但当调用方开始增多, 这些零散差异就会迅速变成维护成本: 客户端要知道该连哪里, 工具脚本要知道某个模型属于哪个 provider, UI 也要重复实现一套又一套连接逻辑。

`lm_relay` 就是在这种背景下产生的. 它做的事情并不复杂: 把上游不同模型服务统一暴露成一个兼容 OpenAI / Anthropic 的入口, 然后只根据 `model` 名称决定请求该转发到哪里。对调用方来说, 它看到的只是一个固定地址; 对中继层来说, 它内部再去处理模型改写、provider 选择和鉴权拼接。这种拆分并不华丽, 但很实用, 因为它把“如何调用模型”重新变成了服务端的责任。

这个项目里我最看重的并不是“转发”本身, 而是路由规则的表达方式. `lm_relay` 的核心映射由 `model_from`、`model_to` 和 `provider_id` 组成, 其中 `model_from` 支持正则, `model_to` 支持替换捕获组。这意味着调用方并不一定要知道真实的上游模型名, 只要遵守一套稳定的逻辑命名即可。例如把 `^cli/(.+)$` 改写成真正的目标模型, 或者把看起来像 `claude-*` 的请求重新导向本地模型。很多时候, 统一命名比统一接口更重要, 因为它真正减少的是上层系统的条件分支。

为了让这个统一入口尽可能少打扰现有生态, `lm_relay` 直接兼容了常见的几类调用方式, 包括 `POST /v1/chat/completions`、`POST /v1/responses`、`POST /v1/messages` 和 `GET /v1/models`。这意味着不管是偏 OpenAI 风格的客户端, 还是偏 Anthropic 风格的客户端, 都不需要围着中继层重写 SDK。中继真正需要做的是在请求进入后补齐鉴权头、改写模型名, 再把流式和非流式响应尽量原样返回出去。它更像一层窄而硬的适配层, 而不是一个企图重新定义交互协议的平台。

这套设计里还有两个我觉得很关键的细节。第一个是对 `LM Studio` 上下文长度的处理。项目没有把 `context_length` 做成一堆抽象配置, 而是直接采用类似 `16k>128k>200k` 的顺序升级策略: 默认先用较小上下文启动, 一旦因为上下文不足报错, 就按预设档位逐步重试。这种设计不是最“优雅”的, 但很符合本地推理的现实, 因为显存、加载时间和可用上下文原本就是相互牵制的。第二个细节是 `cli` provider 的启动方式。第一次命中对应路由时, `lm_relay` 会自动 clone `CLIProxyAPI`, 生成本地 token, 然后通过 `docker compose` 拉起服务。对使用者来说, 这相当于把原本零碎的环境启动动作折叠进了一次真实请求里。

如果只是一个简单的转发器, 这个项目大概不会让我觉得它值得单独写一篇文章。真正让我愿意保留它的是它在“可管理性”上的补充: 除了 `FastAPI` 提供的中继接口, 还有一套 daemon 运行方式和本地控制面板。`lm-relay api start` 会把服务放到后台, 同时把状态写进 `~/.local/state/lm_relay/`; `healthz` 负责回传 PID、运行时长和当前配置路径; 而 `Streamlit` 控制面板则通过本地 `admin/config` 接口读取和更新配置。这里有一个我刻意保守的限制: 配置文件里的 `host` 和 `port` 即使被修改, 也不会对当前 daemon 做热切换, 必须重启服务才能真正生效。这个限制看起来不够“智能”, 但它避免了“配置已经改了, 进程却还监听旧端口”这种非常隐蔽的问题。

`lm_relay` 还有一种我很喜欢的气质: 它没有假装自己是一个庞大的网关系统。它不试图把所有 provider 抽象成同一种能力模型, 也没有引入复杂的插件架构, 而是先认真解决一个更现实的问题: 当我本地已经有几套模型入口, 如何让客户端只记住一个地址。很多工程项目真正的难点都不在算法, 而在边界清不清楚。这个项目的边界相对明确: 统一入口、统一映射、最少的管理功能, 然后把剩下的复杂性留在上游 provider 自己那里。

我越来越觉得, 本地 AI 工具链会持续分裂。模型、运行时、控制台、兼容层都会越来越多, 很难指望某一个平台把所有问题一次性统一掉。与其等待一个完美中枢, 不如先做几个足够薄、足够可靠的中间层。`lm_relay` 就是这样的东西: 它并不试图成为 AI 基础设施的中心, 只是把原本分散而脆弱的调用路径收束成一个可以理解、可以维护、也可以逐步扩展的入口。
