从 AI 聊天衍生出来的 AI 编程工具，都对即时通讯的交互模式情有独钟。从 Codex 和 Claude Code，再到各类 IDE 插件、AI 编程客户端，无一不是让用户在与 AI 的持续对话中完成需求沟通和方案调整。

经典交互流程往往是开发者提出需求，AI 直接开始生成代码。而一些 AI 工具进一步提供 Plan 模式，即在生成代码之前通过提问的方式定下实现方案，再基于方案生成代码。这样的小修小补确实有一定效果，但是并没有完全解决方案修订效率低下的问题。

我认为 Vibe Coding 领域中人机的最佳交互形态，是围绕一份实现方案展开的文档协作。

## Vibe Coding 的循环工作流

一个开发需求 REQ 往往包含多个需要确认的实现细节，我们将每个细节称为一个 SPEC。

每个 SPEC 都要经历三个步骤：开发者编辑方案，AI 分析并修改方案，开发者审查 AI 的修改。等本轮涉及的 SPEC 达成一致后，AI 才根据方案落地代码，开发者再检查运行效果。

~~~text
提出需求 REQ
    ↓
拆分为多个 SPEC
    ↓
协作完善本轮受影响的 SPEC
    ↓
AI 根据方案落地代码
    ↓
开发者检查运行效果
    ├─ 不符合预期：修改受影响的 SPEC，进入下一轮
    └─ 符合预期：结束
~~~

### 开发者时间与端到端时间

AI 的分析时间会计入端到端耗时，但并不等于开发者的主动工作时间。开发者真正投入的核心时间来自编辑每个 SPEC 和审查每个修改结果，此外还要承担切换上下文、重新定位修改位置和合并结果的成本。

串行即时通讯把工作组织成一轮轮“发送—等待—审查”；邮件通讯减少了打断，却将反馈推迟到整个批次之后；文档协作则让开发者保持连续编辑，同时让 AI 在后台消费已经产生的修改。

因此，提高 Vibe Coding 效率并不只是让 AI 更快，而是减少无意义的交接和等待，让开发者的主动工作与 AI 的处理尽可能重叠。

不同协作模式没有减少必须完成的编辑、分析和审查工作，区别在于 AI 分析能否与开发者编辑重叠，以及需要多少次交接和上下文切换。

### 常见 Vibe Coding 协作模式对比

为了保证比较公平，三种协作模式必须处理同一组 SPEC，也必须让每个 SPEC 经历相同数量的沟通轮次。区别只能来自这些工作如何调度，而不能一边假设邮件一次命中，另一边却允许文档持续修订。

使用以下变量描述共同的背景参数：

- \(N\)：REQ 拆分出的 SPEC 数量
- \(R_i\)：第 \(i\) 个 SPEC 达到预期所需的沟通轮数
- \(n_r=\#\{i\mid R_i\ge r\}\)：第 \(r\) 轮仍需沟通的 SPEC 数量，其中 \(n_1=N\)
- \(T_{\text{edit},r}\)：第 \(r\) 轮编辑一个 SPEC 的平均时间；首轮编辑和返工编辑可以取不同值
- \(T_{\text{ai}}\)：AI 分析并修改一个 SPEC 一轮的平均时间
- \(T_{\text{review}}\)：开发者审查一个 SPEC 一轮结果的平均时间
- \(C_{\text{ai}}\)：可同时工作的 AI 或 subagent 数量

单个 SPEC 的平均沟通轮数是一个派生量：

\[
\bar R=\frac{\sum_{r\ge1}n_r}{N}
\]

例如，4 个 SPEC 都要完成首轮，其中 2 个还要进入第二轮，则 \(n_1=4,n_2=2\)，平均轮数恰好是 \((4+2)/4=1.5\)。这和“整个 REQ 有一半概率整批重做一次”不是同一种模型，不能都简化成单轮时间乘以 1.5。

三种协作模式需要完成的开发者主动工作完全相同：

\[
T_{\text{active}}=\sum_{r\ge1}n_r\left(T_{\text{edit},r}+T_{\text{review}}\right)
\]

下面仍然只比较 SPEC 协作阶段。最终代码落地和运行检查在背景参数相同时属于共同成本，因此不计入公式。

#### 即时通讯模式

最常见的 Vibe Coding 交互，是开发者和 AI 在一个会话中串行往返。开发者发送当前 SPEC，等待 AI 完整响应后进行审查；如果结果不符合预期，就继续围绕该 SPEC 沟通，完成后再转向下一个 SPEC。

所有 SPEC-round 的“编辑—AI—审查”都依次落在关键路径上，因此：

\[
T_{\text{instant}}=
\sum_{r\ge1}n_r
\left(T_{\text{edit},r}+T_{\text{ai}}+T_{\text{review}}\right)
\]

这是后续模式的串行比较基线。它的优点是启动成本低、上下文集中，也能很早获得第一个反馈。缺点是同一会话内没有人机重叠，增加 AI 并发数也不会缩短这条串行关键路径。

#### 邮件通讯模式

邮件通讯把仍需处理的 SPEC 组织成一批。开发者编辑完本轮全部内容后发送，AI 才能开始；等整批结果全部返回并审查后，才能组织下一封邮件：

~~~text
第 1 轮：编辑整批 → AI 并行处理整批 → 审查整批
                                      ↓ 仍有 SPEC 未达预期
第 2 轮：编辑返工批次 → AI 并行处理返工批次 → 审查返工批次
~~~

当本轮有 \(n_r\) 个 SPEC、AI 并发数为 \(C_{\text{ai}}\) 时，AI 需要完成 \(\lceil n_r/C_{\text{ai}}\rceil\) 个处理波次。因此邮件通讯的总时间为：

\[
T_{\text{email}}=
\sum_{r\ge1}
\left[
n_r\left(T_{\text{edit},r}+T_{\text{review}}\right)
+\left\lceil\frac{n_r}{C_{\text{ai}}}\right\rceil T_{\text{ai}}
\right]
\]

邮件可以借助 subagent 将整批 AI 墙钟时间压缩到接近一个 SPEC 的处理时间，但每一轮仍然存在“全部编辑完才能发送、全部返回后才能返工”的批次屏障。平均轮数上升时，这个屏障会重复出现。

#### 文档协作模式

文档协作不是把多个 SPEC 攒成一个更大的 Prompt，而是将每个段落的修改变成可以立即调度的增量任务。首轮中，开发者保持连续编辑；首轮编辑完成后，按结果到达顺序审查。某个 SPEC 需要返工时，可以立即修改并再次交给 AI，不必等待其他 SPEC 组成下一批：

~~~text
开发者：编辑 SPEC₁ → 编辑 SPEC₂ → 编辑 SPEC₃ → 审查/返工已返回结果 → ...
AI：             分析 SPEC₁ → 分析 SPEC₂ → 分析 SPEC₃
                                  ↓ SPEC₁ 返工后立即继续分析
~~~

三种协作模式的任务总量相同，但文档协作没有全局轮次屏障。它的总时间不是各阶段时间的简单相加，而是这组任务在“1 名开发者、\(C_{\text{ai}}\) 个 AI 执行槽”上的关键路径长度。

对第 \(i\) 个 SPEC 的第 \(r\) 轮，任务依赖为：

\[
E_{i,r}\rightarrow A_{i,r}\rightarrow V_{i,r}
\quad\text{且}\quad
V_{i,r}\rightarrow E_{i,r+1}
\]

其中编辑 \(E\) 和审查 \(V\) 共用同一个开发者执行槽，分析 \(A\) 使用最早空闲的 AI 执行槽。若一个 AI 任务在 \(d_q\) 时被提交，各 AI 槽当前空闲时间为 \(f_1,\ldots,f_{C_{\text{ai}}}\)，则该任务完成时间为：

\[
c_q=\max\left(d_q,\min_j f_j\right)+T_{\text{ai}}
\]

占用的 AI 槽随后把空闲时间更新为 \(c_q\)。开发者完成首轮连续编辑后，优先审查最早返回的结果。若此时开发者执行槽在 \(h\) 时空闲，则结果 \(q\) 的审查完成时间，以及需要返工时下一轮 AI 任务的提交时间分别为：

\[
v_q=\max(c_q,h)+T_{\text{review}},
\qquad
d_{q'}=v_q+T_{\text{edit},r+1}
\]

开发者执行槽相应更新为 \(v_q\) 或 \(d_{q'}\)。最终总时间定义为每个 SPEC 最后一轮审查完成时间的最大值：

\[
T_{\text{doc}}=\max_i C^{\text{review}}_{i,R_i}
\]

这组递推比一个只计算“尾部等待”的闭式公式更接近实际调度：AI 并发有限、返工出现在哪些 SPEC、开发者是否有其他结果可以继续审查，都会改变关键路径。下方模拟器使用的正是这套离散事件模型。

### 用相同参数动态比较

模拟器默认使用 4 个 SPEC，其中 2 个需要第二轮，即平均每个 SPEC 沟通 1.5 轮。首轮和返工编辑均为 5 分钟，AI 每个 SPEC 每轮需要 8 分钟，审查需要 3 分钟，并允许 4 个 AI 任务并行。

在这个默认场景中，三种协作模式都完成 6 个 SPEC-round，开发者主动工作时间也完全相同。可以调整 SPEC 数量、每轮返工数量、编辑/AI/审查耗时和 AI 并发数，观察批次屏障与增量调度怎样改变端到端耗时。

默认将两个返工 SPEC 均匀放在编辑序列中，模拟结果为：

| 协作模式 | SPEC 协作时间 | 开发者主动时间 | 第一个可审查结果 |
| --- | ---: | ---: | ---: |
| 即时通讯 | 96 分钟 | 48 分钟 | 13 分钟 |
| 邮件通讯 | 64 分钟 | 48 分钟 | 28 分钟 |
| 文档协作 | 50 分钟 | 48 分钟 | 13 分钟 |

返工发生在编辑序列的什么位置，也会改变文档协作的关键路径。模拟器因此提供了“均匀、较早、较晚”三种分布，而不是把这个影响隐藏在平均轮数里。
