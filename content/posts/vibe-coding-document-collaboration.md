从 AI 聊天衍生出来的 AI 编程工具，都对即时沟通的交互模式情有独钟。从 Codex 和 Claude Code，再到各类 IDE 插件、AI 编程客户端，无一不是让用户在与 AI 的持续对话中完成需求沟通和方案调整。

经典交互流程往往是开发者提出需求，AI 直接开始生成代码。而一些 AI 工具进一步提供 Plan 模式，即在生成代码之前通过提问的方式定下实现方案，再基于方案生成代码。这样的小修小补确实有一定效果，但是并没有完全解决方案修订效率低下的问题。

我认为 Vibe Coding 领域中人机的最佳交互形态，是围绕一份实现方案展开的文档协作。

## Vibe Coding 的循环工作流

一个开发需求 REQ 往往包含多个需要确认的实现细节，我们将每个细节称为一个 SPEC。

每个 SPEC 都要经历三个步骤：开发者编辑方案，AI 分析并修改方案，开发者审查 AI 的修改。等本轮涉及的 SPEC 达成一致后，AI 才根据方案落地代码，开发者再检查运行效果。

~~~text
提出需求 REQ
    ↓
拆分为多个 SPEC
    ↓
协作完善本轮受影响的 SPEC
    ↓
AI 根据方案落地代码
    ↓
开发者检查运行效果
    ├─ 不符合预期：修改受影响的 SPEC，进入下一轮
    └─ 符合预期：结束
~~~

为了比较不同协作方式，使用带有明确语义下标的变量：

- \(N_{\text{spec}}\)：本轮需要处理的 SPEC 数量
- \(T_{\text{spec-edit}}\)：开发者编辑一个 SPEC 的平均时间
- \(T_{\text{spec-analyze}}\)：开发者等待 AI 分析并修改一个 SPEC 的平均时间
- \(T_{\text{spec-review}}\)：开发者审查一个 SPEC 修改结果的平均时间

如果所有工作完全串行，那么每个 SPEC 都要依次经历完整的“编辑—分析—审查”过程：

\[
T_{\text{serial-total}}=N_{\text{spec}}\left(T_{\text{spec-edit}}+T_{\text{spec-analyze}}+T_{\text{spec-review}}\right)
\]

首轮需要处理全部 SPEC，后续返工通常只涉及其中一部分，此时只需改变 \(N_{\text{spec}}\) 的取值。代码落地和运行检查在各种协作模式中都会发生，因此下面只比较差异最大的 SPEC 协作阶段。

这个串行时间是后文的比较基线。不同模式没有减少必须完成的编辑、分析和审查工作，区别在于 AI 分析能否与开发者编辑重叠，以及需要多少次交接和上下文切换。

### 开发者时间与端到端时间

AI 的分析时间会计入端到端耗时，但并不等于开发者的主动工作时间。开发者真正投入的核心时间来自编辑每个 SPEC 和审查每个修改结果，此外还要承担切换上下文、重新定位修改位置和合并结果的成本。

即时沟通会把开发者的注意力切成大量“编辑—等待—审查”的短周期；邮件沟通减少了打断，却将反馈推迟到整个批次之后；文档协作则让开发者保持连续编辑，同时让 AI 在后台消费已经产生的修改。

因此，提高 Vibe Coding 效率并不只是让 AI 更快，而是减少无意义的交接和等待，让开发者的主动工作与 AI 的处理尽可能重叠。

### 常见 Vibe Coding 模式对比

为了保证比较公平，下面三种模式处理的是同一个包含多个 SPEC 的 REQ。比较过程中不改变 REQ 的拆分粒度；我们只观察协作方式本身如何安排这些 SPEC，不讨论 AI 内部如何调度任务。

#### 即时沟通模式：每个 SPEC 一个同步点

即时沟通通常按照下面的顺序工作：

~~~text
编辑 SPEC₁ → 等待 AI₁ → 审查 SPEC₁
           → 编辑 SPEC₂ → 等待 AI₂ → 审查 SPEC₂
           → ...
~~~

令 \(T_{\text{instant-context-switch}}\) 表示即时沟通中每处理一个 SPEC 产生的平均交接和上下文恢复时间，则：

\[
T_{\text{instant-total}}=N_{\text{spec}}\left(T_{\text{spec-edit}}+T_{\text{spec-analyze}}+T_{\text{spec-review}}+T_{\text{instant-context-switch}}\right)
\]

开发者必须等当前 SPEC 的 AI 结果返回并完成审查，才能自然地进入下一个 SPEC。人与 AI 基本处于串行状态，因此协作时间接近前面的串行基线，还会额外产生多次交接和上下文切换。

它的优点是第一个 SPEC 可以很快得到反馈，适合只有一两个细节的小任务；缺点是每个 SPEC 都形成一次同步点，所有 AI 分析时间都落在关键路径上。

#### 邮件沟通模式：整个批次一个同步点

邮件沟通允许开发者一次编辑多个 SPEC，但 AI 只有在邮件发送后才能开始处理：

~~~text
编辑 SPEC₁ → 编辑 SPEC₂ → ... → 编辑最后一个 SPEC
                                             ↓ 发送
AI 处理 SPEC₁ → AI 处理 SPEC₂ → ... → 处理最后一个 SPEC
                                                      ↓ 返回
审查 SPEC₁ → 审查 SPEC₂ → ... → 审查最后一个 SPEC
~~~

令 \(T_{\text{email-analyze}}\) 表示开发者发送整批 SPEC 后，等待 AI 返回全部结果的时间；\(T_{\text{email-handoff}}\) 表示整批发送和返回产生的交接时间，则：

\[
T_{\text{email-total}}=N_{\text{spec}}\left(T_{\text{spec-edit}}+T_{\text{spec-review}}\right)+T_{\text{email-analyze}}+T_{\text{email-handoff}}
\]

邮件把 AI 处理时间集中成一个连续的阻塞阶段。AI 无法在开发者编辑后续 SPEC 时提前分析，开发者也必须等整个批次完成后才能开始审查。这里不讨论 AI 内部如何处理这一批 SPEC，只计算开发者实际感知到的整批等待时间。

因此，邮件提高了单次沟通的信息承载量，却牺牲了第一个反馈的到达时间，也没有形成真正的人机流水线。

#### 文档协作模式

文档协作不是把多个 SPEC 攒成一个更大的 Prompt，而是将每个段落的修改变成一个可以立即调度的增量任务：

~~~text
开发者：编辑 SPEC₁ → 编辑 SPEC₂ → 编辑 SPEC₃ → ...
AI：             分析 SPEC₁ → 分析 SPEC₂ → 分析 SPEC₃ → ...
全部分析完成后：审查 SPEC₁ → 审查 SPEC₂ → 审查 SPEC₃ → ...
~~~

开发者完成 SPEC₁ 后，不需要等待 AI 返回，而是继续阅读并编辑 SPEC₂。与此同时，AI 已经可以分析 SPEC₁ 的差异，并持续处理后续到达的修改。

为了便于比较，仍然假设开发者完成本轮全部编辑后，再统一审查 AI 的结果。开发者完成最后一个 SPEC 时，AI 已经处理了一部分先前产生的修改，关键路径上只剩尚未完成的等待时间。

令 \(T_{\text{document-analyze-tail}}\) 表示开发者完成最后一个 SPEC 时仍需等待 AI 分析和修改的时间，\(T_{\text{document-handoff}}\) 表示文档增量修改识别和结果回写产生的交接时间，则：

\[
T_{\text{document-total}}=N_{\text{spec}}\left(T_{\text{spec-edit}}+T_{\text{spec-review}}\right)+T_{\text{document-analyze-tail}}+T_{\text{document-handoff}}
\]

文档协作的优势不在于减少了阅读、编辑和审查每个 SPEC 所必需的认知成本，而在于消除了“必须编辑完全部内容才能发送”的批次屏障，使开发者编辑与 AI 分析形成流水线。

文档协作的实际收益取决于 AI 处理修改的速度。如果 AI 能及时消费已经产生的修改，最后一次编辑后的等待就会较短；如果修改积压，等待就会变长。只有当 SPEC 高度耦合，导致 AI 必须等全部编辑完成后才能开始处理时，文档协作才会逐渐退化为邮件式批处理。对于只有一个 SPEC 的小任务，即时沟通往往拥有更低的启动成本。

### 一个简化的时间示例

假设一轮中包含 4 个 SPEC，并且：

- \(N_{\text{spec}}=4\)
- \(T_{\text{spec-edit}}=5\) 分钟
- \(T_{\text{spec-analyze}}=8\) 分钟
- \(T_{\text{spec-review}}=3\) 分钟
- 暂不计算代码落地、运行检查和交接成本
- 邮件沟通中，\(T_{\text{email-analyze}}=32\) 分钟
- 文档协作中，\(T_{\text{document-analyze-tail}}=17\) 分钟
- 文档协作在本轮 AI 分析全部完成后统一审查结果

那么：

| 模式 | SPEC 协作时间 | 第一个 AI 结果返回时间 |
| --- | ---: | ---: |
| 即时沟通 | 64 分钟 | 13 分钟 |
| 邮件沟通 | 64 分钟 | 52 分钟 |
| 文档协作 | 49 分钟 | 13 分钟 |

在这个算例中，邮件与即时沟通的基础处理总量相同，但邮件的首个反馈更晚；即时沟通还会额外产生多次上下文切换。文档协作利用开发者编辑后续 SPEC 的 15 分钟处理先前修改，将总时间从 64 分钟降至 49 分钟。
